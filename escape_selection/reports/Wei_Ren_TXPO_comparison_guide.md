# Wei Ren の TXPO データと手元のカードを比べるための入口

## 先に結論

このデータは使えます。ただし、手元のカードと開始値を照合するデータではありません。

Wei Ren の各 `txpo46dynamics*` ファイルは、基本的に次の3行です。

1. 巨大な奇数1個の2進表記
2. その整数が1になるまでの I/O 列
3. ビット長、最大ビット長、総ステップ数、I数、O数、O/I比

ファイル名の `1000000` は「100万軌道」ではなく、「出発整数が100万ビット」という意味です。各ビット長につき、軌道は1本です。

したがって、いま最初に比べるべきものは次です。

- Wei Ren の I/O 列を、手元で使っている `k = v2(3n+1)` の列へ戻す
- `k` 1個、連続する `k`、短い `k` ブロックの頻度を調べる
- 手元のカードのうち、同じ観測単位を持つものだけと比べる

## I/O と k の対応

Wei Ren の定義は次です。

- `I`：奇数で `(3x+1)/2`
- `O`：偶数で `x/2`

1回の `I` と、その直後に続く `O` をまとめると、odd-accelerated Collatz の1ステップになります。

```text
I          -> k=1
IO         -> k=2
IOO        -> k=3
IOOO       -> k=4
```

つまり、`k = 1 + I直後の連続O数` です。

## 今回の変換結果

全10ファイルで、元ファイルの I数・O数・総ステップ数と復元結果が一致しました。

| 開始値のビット長 | I/O総数 | 復元したk数 | O/I | k=1の比率 | 111窓の比率 | 幾何分布との差（TV） |
|---:|---:|---:|---:|---:|---:|---:|
| 1,000 | 4,636 | 2,294 | 1.020924 | 0.510462 | 0.139180 | 0.022747 |
| 10,000 | 49,017 | 24,617 | 0.991185 | 0.502661 | 0.127564 | 0.004485 |
| 100,000 | 485,260 | 243,072 | 0.996363 | 0.501300 | 0.126437 | 0.002471 |
| 1,000,000 | 4,812,415 | 2,405,366 | 1.000700 | 0.499807 | 0.124602 | 0.000373 |
| 6,000,000 | 28,911,397 | 14,455,482 | 1.000030 | 0.500081 | 0.125041 | 0.000135 |

ここでの「111窓」は、連続する3個の `k` が `1,1,1` だった窓の無条件比率です。手元の Paradoxical Sequence カードにある `pre111` は特定境界で条件を付けた値なので、数値をそのまま横に並べて同一視してはいけません。

大きいビット長では、無条件の頻度はおおむね次へ近づいています。

```text
P(k=1)=1/2
P(k=2)=1/4
P(k=3)=1/8
...
```

これは最初の基準線です。「傷がない」という意味ではありません。局所条件や境界条件を混ぜる前の、全体平均に近い眺めです。

## 9枚のカードとの接続順

| カード | 今すぐ比べられるか | 理由 |
|---|---|---|
| Collatz finite-block | 最優先 | 観測の中心が valuation word、つまり `k` の列だから |
| Prefix-side descriptors | 次に可能 | `k` のprefixと有限未来を作れるから。ただしWei側は軌道1本 |
| Paradoxical Sequence | 基準線だけ可能 | `k=1` や `1,1,1` は数えられるが、`64-95 -> 32-63` 条件がまだ付いていない |
| Coordinate Audit | 一部可能 | sampled long-word側との比較候補。RT550との開始値joinではない |
| Delta localization | まだ早い | actualとiidを同じセルへ置く定義と重みが必要 |
| Waiting Hall | まだ不可 | `remaining_K`、miss、event keyの再構成が必要 |
| Boundary-front geometry | まだ不可 | band stage、front、chain statusの定義が必要 |
| Finite state-space | まだ不可 | 複合state cardをWei側で再生成する定義が必要 |
| Co-change bundle | まだ不可 | state cardができた後の矢印比較だから |

## 最初の比較で守ること

Wei Ren 側の局所窓は大量にありますが、独立した大量の軌道ではありません。1本の軌道から重なりながら切った窓です。そのため、行数をそのまま「標本数」として扱いません。

また、公開Cコードには再生成上の注意があります。

- `RUNTIMES=1` なので各ファイルは1軌道
- `srand(time(NULL))` がコメントアウトされ、乱数seedが明示記録されていない
- `rand()` の列は処理系に依存する
- バッファ初期化で `sizeof(pointer)` を使っており、コードをそのまま再コンパイルする場合は修正が必要

ただし、開始整数の2進列そのものが各データファイルに保存されています。比較では「同じ乱数を再生成する」必要はなく、保存済みの開始値と I/O 列を入力として使うのが安全です。

## 生成した比較用ファイル

- `wei_ren_txpo_summary.csv`：各巨大軌道の検算と要約
- `wei_ren_k_frequency.csv`：各ビット長でのk頻度と `2^-k` との差
- `wei_ren_k_transition.csv`：連続する `k_t -> k_{t+1}` の頻度
- `wei_ren_top_k_blocks.csv`：長さ3・4の上位kブロック

次の実比較では、まず `Collatz-finite-block` の短い valuation word 表と、`wei_ren_top_k_blocks.csv` を同じ `word=(k1,k2,k3,k4)` キーで接続します。ただし、母集団と正規化が違うため、「一致・不一致」ではなく「Wei Ren の巨大軌道でも同方向に偏るか」という外部照合として扱います。
